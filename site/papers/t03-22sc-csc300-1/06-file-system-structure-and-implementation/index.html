File System Structure Observation

In order for a computing system to be useful in today’s world, it must be able to store data for future use. In the “Filesystem Hierarchy Standard”, LSB Workgroup (2015) specifically menton that a well-formed, standardized filesystem is essential for both the developer and user experience to thrive (p. 1). This paper is an exploration of one such system, and the implications it creates.

Theory

This practice has highlighted a few noteworthy facts about present-day GNU/Linux Distributions, and about present-day operating systems in general. These are explored here.

Standard Filesystem Directories

Briefly, the contents and function for each of the directories listed in the output mentioned above:

  • • / - The root of the filesystem. This is the start of the system, the place from which everything else can be referenced. The following are all directories that are required by standard (aside from /proc/ and /sys/) to exist directly underneath the root filesystem, regardless of OS (LSB Workgroup, 2015, p. 4).

  • • /bin - Binaries that the system needs to function. Core Utilities like sh, ls, and which are often here. By Standard it cannot have any subdirectories inside of it (LSB Workgroup, 2015, p. 5). On Kali Linux, this is a symlink to /usr/bin, so all user binaries are in one place.

  • • /boot - Files the operating system provides to the bootloader to allow for it to be booted properly. The kernel lives here, along with any bootloader configurations. The standard specifically mentions that only files needed at boot time should be included here: Other configuration files for the bootloader or OS can go in /etc instead. And should binaries be required, they should exist in /sbin (LSB Workgroup, 2015, p. 7).

  • • /dev - Linux creates a virtual device identifier for each piece of hardware that exists, or is considered to possibly exist. This is where hard drives live, and the random number generator, and all of the TTYs.

  • • /etc - System-wide configuration files live here. This is where the Filesystem table (fstab), Mount table (mtab), and hostname files live. In the standard, LSB Workgroup (2015) suggest that configuration files in /etc be stored in dedicated subdirectories rather than as a flat collection (p.7).

  • • /home - User files all typically go under this directory, in a folder bearing the same name as the user that will own that folder. Anything can be here, but most often it is Documents, Pictures, Videos, and User Config Files. It is interesting to note that this directory is considered optional by the Filesystem Hierarchy Standard (LSB Workgroup, 2015, p. 10).

  • • /initrd.img - There are four files that I was surprised to see here. The initrd is the “Initial Ram Disk”, which is a collection of drivers needed for the operating system to be successfully booted on the hardware it is running on. Usually this lives under /boot, and indeed, this is merely a symlink to the real Initial Ram Disk in that directory on Kali Linux. I believe this placement is required by U-Boot; perhaps this is why they are here? By Standard, no application should ever create special files or subdirectories under /, so this is likely a system administration decision, and exempt from the standard (LSB Workgroup, 2015, pp. 3–4).

  • • /initrd.img.old - Like the above, this is a symlink to an older copy of the Initial Ram Disk, likely to enable booting after a botched update.

  • • /lib - All of the /lib* directories on Kali Linux are symlinks to their /usr/lib* counterparts. This practice intentionally makes all libraries, core or add-on, available in the same location…But loses the granularity one might have in case of a misbehaving package. I digress, though: /lib is the directory that core libraries for the current architecture live, and where all kernel modules should be (LSB Workgroup, 2015, p. 11).

  • • /lib32 - Due to the slow migration computers made from x86 to amd64 over the last decade and a half, many distributions separate out libraries into directories based on their word size. This is especially important for multilib distributions—that is, distributions that have support for running both 32-bit and 64-bit ELF files. This directory stores the explicitly-32-bit shared libraries.

  • • /lib64 - And this one stores the explicitly-64-bit libraries. This folder exists very commonly, as there are many applications that default to it to provide a clean space for any 32-bit libraries to live. However, on Kali Linux, only the loader library itself lives in this directory. In the FHS, LSB Workgroup (2015) suggest using both /lib32 and /lib64, and creating /lib as a symlink to one of them, if on a multilib system (p. 11).

  • • /libx32 - This is an odd directory, which seems to be pointing to multilib-32-bit libraries, as would be in either /lib or /lib32. The other possibility is cross-compilation libraries, for instance for compiling a program for ARM on x86. But then why the 32?

  • • /lost+found - Any version of the “Extended File System”—ext2, ext3, and ext4 for instance—uses this directory to store data that is obsolete or damaged. The fsck command usually will put files here if problem files are found, and the names of those files are usually their inode number. This directory is therefore not in the FHS (LSB Workgroup, 2015).

  • • /media - This is one of the two places distributions put removable media filesystems when they are mounted and the one recommended for such use by the Standard (LSB Workgroup, 2015, p. 12). Some distributions prefer using /run//media.

  • • /mnt - This is the traditional place for external filesystems to be mounted. This sometimes contains some empty folders with suggestive names for mounting, like /mnt/cdrom or /mnt/tmp, however LSB Workgroup (2015) note that this conflicts with the older practice of using /mnt itself as a mount point.

  • • /opt - This directory generally houses self-contained packages of software, like some Java libraries or TeXLive. It is for software that is considered optional to, and separate from, the system—a practice that goes all of the way back to System V in 1990 (LSB Workgroup, 2015, p. 13).

  • • /proc - This is the traditional place that running process information is accessible to the users. It can be thought of as a “system directory”, in that it is a directory that represents the running system.

  • • /root - This is the administration account’s home directory, where files devoted to their work in that capacity might go. It is usually set up exactly like a user home directory, with the caveat that it usually remains much more empty (as doing arbitrary things as root is discouraged in most distributions). An interesting part of the Standard is that this, too, is optional—and the root directory itself, /, is recommended as a fallback (LSB Workgroup, 2015, p. 14).

  • • /run - This is the other place many distributions put external mounted filesystems, as well as the place where user sockets and daemon files might go. Generally, the init system lives here, as well as any container daemons, udev, and any process identifier files (LSB Workgroup, 2015, p. 14)

  • • /sbin - These are executable programs that only the administrator (or someone with sudo permissions) can access. This is where tools to reboot the system, create and check filesystems, and change user permissions usually live.

  • • /srv - These are directories that are being served to the world through one protocol or another. The Standard defines this as data for services the system provides (LSB Workgroup, 2015, p. 16). If You run a web server, the website itself is probably under this directory. Other candidates include Gemini Nodes and Samba Shares.

  • • /sys - This is the “system directory” created by the Linux Kernel team, and is the one that they recommend all new programs use instead of /proc. It stores the same kind of information as proc, but in a different, more structured way. It did not exist in UNIX, and is Linux-specific.

  • • /tmp - This directory stores files that do not need to be preserved, but are needed for a program to function. This is where a lot of temporary output occurs, where many packages are built, and where You might find some programs place their backup files during execution. This is actually a specific part of the POSIX standard, as well (LSB Workgroup, 2015, p. 17).

  • • /usr - This directory used to store user-installed packages for the system, however nowadays it is more likely to contain distribution-specific packages (and user installed packages instead go in /usr/local). This can be anything from executables in /usr/bin to libraries in /usr/lib to files that need to be accessed by many programs, possibly of different architectures in /usr/share. The /usr/local directory is required by standard to be empty post-installation of a conforming OS (LSB Workgroup, 2015, p. 18).

  • • /var - This directory stores data that changes very often, and so is likely used to cache downloads and as the target for logs. The reason this folder was included in the standard is to enable the /usr hierarchy to be mounted as read-only (LSB Workgroup, 2015, p. 19).

  • • /vmlinuz - This is another file that normally lives under boot; This is the kernel executable itself. It is, again, a symlink to the appropriate file in /boot on Kali Linux.

  • • /vmlinuz.old - And this is the previous kernel, in case there is an issue with an update.

Seven Basic Operations

In their book, Silberschatz et al. (2018) emphasize the breakdown of file operations into seven basics: “Creating”, “Opening”, “Writing”, “Reading”, “Deleting”, “Truncating”, and “Repositioning” (p. 532).

“Creating” a file is when space and a parent directory are allocated for a file. Without creation, a file would never enter the filesystem: Instead, it would live in volatile memory and be lost on reboot.

“Opening” a file is when a file is loaded into memory, marked as open, and assigned a handle. This is fundamental to most operations, as modern operating systems require this for everything but creation and deletion (Silberschatz et al., 2018, p. 533)

It’s interesting to note that “Opening” a file is listed as a basic operation here; It is not included in the traditional 6 basic file operations (Skoglund, 2020, Section 2: Files and Directories). This inclusion is likely a good one, however, as it is today a fundamental operation that, with its inclusion, prevents multiple accesses to the same file in sequence from creating processing overhead.

“Writing” a file takes an open file and dumps the content that currently lives in memory into the file on the file system.

“Reading” a file takes an open file and dumps some or all of the content that currently lives inside of it into memory.

“Repositioning” is when the “current spot” in a file (according to the process doing the operation) is changed, without altering the file or the data in memory. This is also known as “seeking”.

“Deleting” a file is the inverse of “Creating” a file: The space is freed, and the parent directory’s entry for that file is deleted. Two types of links exist that can change these two operations:

“Hard links” allow multiple directory entries per file-space, effectively saving space by having two “files” share the same allocated space on the hard drive. In this case, the space will not be freed until the last entry referring to it is removed. And “symbolic links” are, essentially, pointers to some other virtual file in the Unix system: They have no space devoted to them, and there is no guarantee that they point to a valid allocated space. Deleting a symbolic link will never cause data to be deleted, because it does not actually refer to the data: It refers to the thing that refers to the data (Silberschatz et al., 2018, p. 548).

The last of the basic operations is “Truncating”, where the data is jettisoned (length set to 0) but all other aspects of the file (permissions, directory entry, etc) is kept (Silberschatz et al., 2018, p. 532).

File Types

File types allow the operating system to enforce (or encourage) the use of a file in a specific way, and so can help the user avoid trying to view a text file as an image, or “play” an executable file as though it were an audio file, both of which can produce undefined behavior (Silberschatz et al., 2018, p. 536).

There are two main approaches: File extensions, where a (usually three-character, sometimes less or more) suffix is added to a file name after a “.” (and sometimes creation information is embedded in an attribute as well, so a can easily be opened by the program that created it). And Unix’s “magic numbers”, which are embedded in the file itself, sparsely implemented, and lazily enforced (Silberschatz et al., 2018, p. 537). Whether this is a boon or a bane is an exercise for the reader, but either way both approaches have advantages and disadvantages, but the file extension is more often used because it is more visible to the average end user, and easier to extend.
